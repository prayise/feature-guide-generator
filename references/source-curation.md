# 소스 큐레이션 전략

## 왜 큐레이션이 중요한가

"좋은 슬라이드 = 좋은 프롬프트 × **좋은 소스**". 실제로 결과 품질의 70%는 소스에서 결정된다. 나머지 30%가 프롬프트.

## 3종 소스 카테고리

| 카테고리 | 수량 | 역할 |
|---------|:---:|------|
| **정책 문서** | 1~2장 | Single source of truth — 근거 및 인용 원본 |
| **구현 코드** | 5~10개 | 정책이 실제 어떻게 동작하는지 검증 |
| **화면 이미지** | 5~15장 | 추상 설명을 구체적 UI 영역으로 앵커링 |

## 카테고리별 선별 기준

### A. 정책 문서

**넣을 것**:
- 상세기획서 / PRD / 정책 정리 문서 (`*--detailed-spec.md`)
- 상위기획서 (전략 맥락이 필요한 경우)
- 요구사항 문서(`*--requirements.md`, 원본 근거 확인용)

**빼야 할 것**:
- 회의록 / 피드백 로그 — 중복·노이즈
- 과거 버전 기획서 — 혼란 유발

### B. 구현 코드

**균형 원칙**: API + 데이터 + UI 3축을 섞는다.

| 축 | 대표 파일 패턴 | 역할 |
|----|--------------|------|
| API 핸들러 | `*-handler.ts`, `*-controller.ts`, `*-service.ts` | 비즈니스 로직 / 밸리데이션 / 에러 처리 |
| 스키마 / 엔티티 | `*schema.ts`, `*entity.ts`, `*-dto.ts` | 데이터 구조 / 필드 제약 |
| UI 뷰 | `*.tsx` in `views/`, `components/` | 실제 사용자 인터랙션 |

**선별 기준**:
- 파일 크기 100~500줄 (너무 작으면 컨텍스트 부족, 너무 크면 노이즈)
- 해당 기능의 '주요 경로' 파일 (테스트 파일, 유틸, 재사용 컴포넌트는 제외)
- 관련성 명확한 것만 — 간접 연관은 슬라이드 품질 저하

**자동 추출 패턴** (Claude Code에서):
```
1. 프로젝트 context.md 읽기 → 도메인 키워드 추출
2. Grep/Glob으로 키워드 매칭 파일 나열
3. 파일 크기와 파일명 기준으로 1차 필터
4. 각 파일 상단 30줄 확인 → 관련성 최종 확인
5. 5~10개 선택
```

### C. 화면 이미지

**넣을 것**:
- 핵심 기능 화면 기본 상태 (default)
- 같은 화면의 상태별 변형 (검색·빈 상태·에러 상태)
- 인터랙션 플로우 (A→B→C 순서의 연속 화면)
- 목업이든 실제 스크린샷이든 무관

**빼야 할 것**:
- 다른 기능의 관련 없는 화면
- 해상도 낮거나 잘린 이미지
- 디자인 초기안 — 현재와 다를 수 있음

**필수 포함 패턴**:
- 기본 / 검색 / 빈 상태 3종 세트 → 빈 상태 처리 설명에 필수
- Before / After 쌍 → 변경점 설명에 결정적

## 교차 검증 효과

세 소스가 만날 때만 나오는 슬라이드 품질 예시:

| 질문 | 정책만 | 정책+코드 | 정책+코드+화면 |
|------|-------|----------|---------------|
| "관리 권한 OFF 시 어떻게 되나?" | "메모리 보존" (추상) | "React state 보존, DB 미반영" (구체적이나 UI 불명) | 토글 OFF 화면 + "옵션 영역 숨김, 재ON 시 복원" (완전함) |
| "빈 상태는 어떻게 표시되나?" | 설명만 있음 | 조건부 렌더링 코드 | 실제 빈 상태 화면 + 정확한 문구 |
| "권한 변경 시 즉시 적용?" | "즉시 반영" | `onChange` 핸들러에서 API 호출 | 토스트 피드백 화면 + 실패 시 롤백 화면 |

## 함정 대응

### 함정 1: `.ts/.tsx` 업로드 거부

NotebookLM은 TypeScript MIME 타입을 400으로 거부한다. 마크다운 래핑 필수.

### 함정 2: 너무 많은 소스

15개 넘어가면:
- 색인 시간 증가 (기다려야 함)
- 슬라이드 생성 시 관련성 낮은 인용 등장
- 프롬프트에서 어떤 소스를 참조할지 모호해짐

→ **10개 내외로 제한**. "꼭 필요한 것만".

### 함정 3: 이미지 파일명이 영문·한글 섞임

NotebookLM 내부 처리에서 한글 파일명이 깨지는 경우가 있다. 프롬프트에서 이미지를 참조할 때 **정확한 원본 파일명**을 명시한다 (한글 포함 OK, 단 일관성).

### 함정 4: 실패 소스 잔여물

업로드 실패한 소스가 `preparing`으로 남으면 슬라이드 생성 시 참조 대상에 포함되어 품질 저하. 생성 전 반드시 정리:

```bash
notebooklm source list --notebook "$NBID"
# preparing 상태 소스 찾아서:
yes y | notebooklm source delete <id> --notebook "$NBID"
```

## 큐레이션 빠른 워크플로우

```
[1] 정책 문서 1장 (projects/.../detailed-spec.md)
    ↓
[2] context.md 읽고 도메인 키워드 3~5개 추출
    ↓
[3] Grep/Glob으로 API·Web 레포에서 관련 파일 스캔
    ↓
[4] 파일 크기·이름으로 1차 필터 → 5~10개 선별
    ↓
[5] .ts/.tsx는 마크다운 래핑
    ↓
[6] mockup/assets에서 핵심 화면 5~15장 선별
    ↓
[7] 동일 화면의 상태별 변형(기본/검색/빈) 포함
    ↓
[8] 전체 15~25개 소스를 NotebookLM에 업로드
    ↓
[9] 모두 ready 확인 → 실패분 정리
    ↓
[10] 생성 단계로 진행
```
