# 프롬프트 설계 템플릿 & 예시

## 모드 확인 먼저

프롬프트를 쓰기 전에 어떤 모드인지 먼저 결정한다. NotebookLM은 **이미지를 생성하지 않는다** — 이미지가 없으면 텍스트 모드로, 있으면 이미지 기반 모드로 프롬프트가 달라진다.

| 모드 | 이미지 소스 | 슬라이드 구성 프롬프트 방식 |
|------|-----------|--------------------------|
| **A. 정책·코드 모드** | 없음 | 텍스트·도식 중심. "화면 이미지" 언급 **금지** (없는 이미지를 참조하려 함). 대신 "코드 구조를 기반으로 화면 레이아웃을 텍스트 다이어그램(ASCII 박스 등)으로 설명" 지시 |
| **B. 화면별 상세 모드** | PNG 5~15장 | 각 섹션 헤더에 **이미지 파일명 명시** (`[화면 1] — 이미지: filename.png`). 슬라이드 상단에 실제 이미지 배치되도록 지시 |

## 4요소 필수 체크리스트

```
대상: [역할 + 기술 수준]
목적: [이 슬라이드로 달성하고 싶은 것]

슬라이드 구성 (번호·제목 명시):
1. 표지
[섹션 1: XXX]
2. ...
N+1. Q&A

작성 원칙:
- 문구 기준: [정책 원문 인용 / 자연스러운 재서술 / 기술 용어 허용 여부]
- 시각 요소: [이미지 위치 / 강조 박스 / 번호 레이블 스타일]
- 분량: [각 슬라이드 텍스트 밀도]
```

## 케이스별 템플릿

### A. 운영팀 교육 자료 (정책 중심)

```
대상: [팀명] 운영팀 (CSM·교담자·CS 등 비개발 이해관계자)
목적: [기능명] [릴리즈일자] 릴리즈를 고객사에 안내할 수 있도록 정책과 동작을 시각화

슬라이드 구성 (총 15~20장):
1. 표지
2. 무엇이 바뀌는가 — AS-IS vs TO-BE 한 눈에 보기
3. [핵심 개념 1]
...
N. 자주 묻는 Q&A (운영팀 FAQ)
N+1. 릴리즈 이후 문의 처리 프로세스

작성 원칙:
- 문구는 정책 문서 원문 그대로 인용
- 각 슬라이드에 '운영팀 주의사항' 박스
- 기술 용어 대신 '교담자가 화면에서 보게 될 라벨'로 설명
- 코드 파일은 '실제 동작 확인' 필요한 엣지 케이스에서만 인용
```

### B. 화면별 상세 가이드 (UI 중심)

```
대상: [팀명] 운영팀 — 고객사 교담자를 교육·지원
목적: [메뉴명]의 실제 화면으로 개별 기능을 상세히 설명하는 운영팀용 화면 가이드

슬라이드 구성 (총 20~25장):
1. 표지
2. 진입 경로 다이어그램

[화면 1: XXX] — 이미지: {filename1.png}
3. 화면 전체 레이아웃
4. 영역별 상세 ①②③ 번호 레이블
5. 주요 인터랙션

[화면 2: YYY] — 이미지: {filename2.png}
6. ...

[종합]
N. 저장/반영 정책
N+1. 자동 전환 로직
N+2. 운영팀 체크리스트

작성 원칙:
- 각 슬라이드 상단에 화면 이미지를 큼직하게 배치
- 하단에 번호가 매겨진 레이블(①②③)로 영역별 설명
- 문구는 정책 문서 원문 인용
- '운영팀 TIP' 박스로 고객 Q&A 대응 포인트 강조
- 기술 용어 대신 실제 UI 라벨 우선
```

### C. 고객사 온보딩 자료 (셀프러닝용)

```
대상: 고객사 교육담당자 — 기능 최초 사용자, 기술 비전문가
목적: 셀프 온보딩할 수 있도록 첫 사용 플로우를 단계별로 안내

슬라이드 구성 (총 15~20장):
1. 표지 — 환영 메시지
2. 이 서비스로 할 수 있는 것 (가치 제안)
3. 시작 전 준비사항

[Step-by-Step 플로우]
4. Step 1: 로그인
5. Step 2: [첫 번째 액션]
...
N. Step N: [완료]

[상황별 대응]
N+1. 자주 하는 실수
N+2. 막혔을 때 — 도움받는 방법

작성 원칙:
- 기술 용어 최소화, 친근한 톤
- 각 Step에 화면 스크린샷 + 강조 표시(화살표·빨간 박스)
- '완료 후 확인' 체크 포인트 명시
- 에러 상황은 '이렇게 하면 됩니다' 식의 해결형 프레이밍
```

### D. 사내 기술 공유 (엔지니어 대상)

```
대상: 사내 개발자 — 같은 스택 경험자, 아키텍처 관심 있음
목적: [기능명]의 설계 결정과 구현 핵심을 공유하고 팀 논의 촉발

슬라이드 구성 (총 20~25장):
1. 표지 — [기능명]: 어떻게 만들었고 무엇을 배웠나
2. 요구사항 & 제약조건
3. 아키텍처 전체 구조

[설계 결정]
4. 결정 1: [choice A vs B vs C] → 왜 B를 선택했나
5. 결정 2: ...

[구현 디테일]
N. 핵심 모듈 [파일명]
N+1. 트리키한 엣지 케이스

[회고]
M. 잘된 점 / 아쉬운 점
M+1. 다음에 하면 다르게 할 것

작성 원칙:
- 코드 스니펫 일부 인용 가능 (Before/After 비교 등)
- 파일 경로는 정확하게 명시
- 과도한 추상 설명보다 "실제 부딪힌 문제"를 구체적으로
- 질문을 던지는 슬라이드 1~2장 포함 (토론 유도)
```

### E. CS 매뉴얼 (대응 플레이북)

```
대상: CS팀 — 티켓 응대, 1차 고객 접촉
목적: 자주 들어오는 문의 패턴별로 즉시 꺼내 쓸 수 있는 대응 플레이북

슬라이드 구성 (총 15~25장):
1. 표지 — 사용법 안내
2. 문의 유형별 인덱스 (빠른 이동 가이드)

[문의 유형 1]
3. 증상 — 고객이 이렇게 말합니다
4. 원인 진단 — 확인 순서
5. 해결 가이드 — 이렇게 답변하세요

[문의 유형 2]
6. ...

[에스컬레이션]
N. 이럴 때는 개발팀에 넘기세요

작성 원칙:
- 각 유형 3장 세트 고정 (증상/진단/해결)
- 고객 원문 예시 인용 ("이거 왜 안 돼요?" 같은)
- 스크립트 템플릿 제공 (토시 그대로 쓸 수 있게)
- 에러 메시지·로그 패턴을 시각적으로 구분
```

## 프롬프트 작성 시 흔한 실수

| 실수 | 증상 | 수정 |
|------|------|------|
| "좋은 슬라이드 만들어줘" 수준의 모호함 | 일반론적 내용 | 구체적 슬라이드 번호·제목 명시 |
| 대상자 누락 | 기술 용어 남용 또는 과도한 설명 | "[대상]의 [기술 수준]" 명시 |
| 이미지 파일명 미언급 | 이미지가 엉뚱한 슬라이드에 들어감 | "[화면 1] — 이미지: filename.png" 식으로 매칭 |
| 문구 기준 미지정 | 원문과 동떨어진 재서술 | "정책 문서 원문 인용" 또는 "재서술 허용" 명시 |
| 너무 많은 슬라이드 요구 | 15분 넘게 걸리거나 실패 | 25장 이내로 제한 |

## 프롬프트 전송 방법 (셸 이스케이프 주의)

긴 프롬프트는 heredoc으로 전달:

```bash
PROMPT=$(cat <<'EOF'
대상: ...
목적: ...
...
EOF
)

notebooklm generate slide-deck --notebook "$NBID" --format detailed --language ko --json "$PROMPT"
```

**주의**: `<<'EOF'` (따옴표 있음) — 쉘 확장 완전 비활성화. 따옴표 없는 `<<EOF`는 `$`, 백틱이 해석됨.
